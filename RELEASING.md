# Releasing (Android APK via GitHub Releases / Obtainium)

This app is distributed to direct-download users through **GitHub Releases**:
every version tag builds a release-signed APK in CI and attaches it (plus its
SHA-256) to a Release. [Obtainium](https://github.com/ImranR98/Obtainium) reads
those Releases to install and auto-update the app ‚Äî no Google account, no Play
Services. The workflow is [`.github/workflows/release.yml`](.github/workflows/release.yml).

> This GitHub/Obtainium release key is **separate** from the Apple App Store /
> Google Play signing keys. That is intentional: different channels have
> different signatures, so a phone installed from Play cannot update from
> Obtainium (and vice-versa) ‚Äî each channel updates within itself. This repo
> only concerns the Obtainium/GitHub channel.

---

## One-time keystore setup

The release APK is signed with a private keystore that lives **only** as GitHub
Actions secrets ‚Äî never in this repository (the whole `android/` tree is
git-ignored and regenerated by `expo prebuild`).

> [!IMPORTANT]
> **This keystore is the app's permanent identity.** If you lose it (or its
> passwords), you can never publish an update existing Obtainium users can
> install ‚Äî they would have to uninstall and reinstall. Generate it once, back
> it up somewhere safe and encrypted, and store the passwords in a password
> manager.

### 1. Generate the keystore (once, locally)

```bash
keytool -genkeypair -v \
  -keystore jouwprivacy-release.keystore \
  -storetype PKCS12 \
  -alias jouwprivacy \
  -keyalg RSA -keysize 4096 -validity 10000
# Choose a strong store password (PKCS12 uses the same value for the key).
# Common Name / org fields can be "JouwPrivacy"; they are not user-facing.
```

Keep `jouwprivacy-release.keystore` out of the repo. Back it up (e.g. encrypted
in your password manager or an offline drive).

### 2. Add the four GitHub Actions secrets

Repo ‚Üí **Settings ‚Üí Secrets and variables ‚Üí Actions ‚Üí New repository secret**,
or with the `gh` CLI:

```bash
# from the folder containing jouwprivacy-release.keystore
base64 -i jouwprivacy-release.keystore | gh secret set ANDROID_KEYSTORE_BASE64 \
  --repo koenschauwaert/app-jouwprivacy.com

gh secret set ANDROID_KEYSTORE_PASSWORD --repo koenschauwaert/app-jouwprivacy.com   # the store password
gh secret set ANDROID_KEY_ALIAS         --repo koenschauwaert/app-jouwprivacy.com   # jouwprivacy
gh secret set ANDROID_KEY_PASSWORD      --repo koenschauwaert/app-jouwprivacy.com   # same as store password for PKCS12
```

| Secret | Value |
| --- | --- |
| `ANDROID_KEYSTORE_BASE64` | base64 of `jouwprivacy-release.keystore` |
| `ANDROID_KEYSTORE_PASSWORD` | the keystore/store password |
| `ANDROID_KEY_ALIAS` | `jouwprivacy` |
| `ANDROID_KEY_PASSWORD` | the key password (same as store for PKCS12) |

The workflow fails fast with a clear message if `ANDROID_KEYSTORE_BASE64` is
missing, and aborts if the produced APK is unsigned or debug-signed ‚Äî so a
mis-set secret can never publish a wrongly-signed release.

## Cutting a release

The marketing version (`expo.version` in `app.json`) is the **single shared
anchor** across iOS, Google Play, and this GitHub / Obtainium channel. The
stores' `versionCode` / `buildNumber` are owned by EAS (remote auto-increment);
this channel's APK `versionCode` is **derived from `expo.version`** in CI, so it
stays monotonic and in sync automatically ‚Äî you never bump it by hand.

### Recommended: `./build.sh github`

From the repo root, `./build.sh github` reuses the **same version-bump prompt as
the store builds**, then tags `v<expo.version>` to fire this workflow. Because it
shares that prompt, iOS, Play, and Obtainium all ship the same version and tag.
It only pushes the tag once the commit is already on the (secret-scanned) public
remote, so it never drags an unscanned commit to the public repo.

### Manual equivalent

1. Set `expo.version` in `app.json` to the release version (e.g. `1.0.3`) and
   commit it. **Do not** touch `versionCode` / `buildNumber` ‚Äî EAS owns the store
   ones, and CI derives the APK one from the version.
2. Push the commit through the secret scan: `./git-sync.sh push`.
3. Tag it ‚Äî the tag must equal `v<expo.version>` ‚Äî and push the tag:
   ```bash
   git -C app tag v1.0.3
   git -C app push origin v1.0.3
   ```

Either way, CI builds the signed APK, verifies the signature, derives the
`versionCode`, and publishes a Release named `JouwPrivacy 1.0.3` with
`jouwprivacy-1.0.3.apk` + `.sha256` attached; Obtainium offers it as an update
automatically.

To rebuild/re-publish an existing tag (e.g. after fixing the workflow), run the
**Release APK** workflow manually from the Actions tab and pass the tag; it will
re-upload the assets to the existing Release.

## Verifying a release

Each Release ships the APK and its SHA-256 so anyone can confirm the download and
diff it against a from-source build. See [`VERIFYING.md`](VERIFYING.md).
